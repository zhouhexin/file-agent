## 1. 用户要求

普通用户不关心系统执行了哪些 Skill、Tool、LangGraph 节点或数据库任务。用户只需要：

1. 在聊天页上传一个或多个文件，并同时输入自然语言任务；当前阶段有附件时也不允许空文字提交。
2. 系统自动保护原件，读取正文，必要时 OCR，并生成可用工作副本。
3. 系统在工作副本首次对用户可见前自动生成标准化文件名和多标签分类建议。
4. 系统保存年份、文档类型、关键词、实体、摘要和可定位原文证据，以便以后检索。
5. 用户通过“找去年奖学金材料”“打开学生工作处的通知”“这份文件讲了什么”等对话访问文件。
6. 普通用户只看到整理后的文件名、分类、概览、引用、失败或确实需要决策的事项。
7. Skill、Tool、AgentRun、ToolInvocation、ChangeSet、内部队列和模型细节只作为后端安全与审计边界，
   不得成为普通用户的使用概念。

后续任何功能计划都必须先回答：它是否直接改善“上传后自动整理”和“通过对话访问文件”。如果不能，
除非它是阻止数据损坏、越权或虚假执行的 P0 安全工作，否则不得排在该闭环之前。

## 2. 产品边界

### 2.1 首次上传整理可以自动执行

允许归档的上传文件完成查重和安全检查后，系统在隐藏临时工作副本内自动执行：

```text
解析/OCR
→ 普通文档摘要
→ 分类主题摘要
→ 年份、文号、标题、单位、关键词、实体提取
→ 多标签分类建议
→ 首次标准化命名
→ 检索索引
→ 原子提交 ACTIVE 工作副本
```

其中普通文档摘要和分类主题摘要默认由本地 CPU-only `Jieba + LexRank` 抽取式 Provider 生成；全局
LLM 开启不能隐式改变该行为。只有用户明确提出总结/讲解时才使用独立的聊天摘要 LLM Provider，或由
部署人员在获得正文处理授权后显式把某个后台摘要 Provider 改为 `llm`。

首次命名属于创建工作副本的内部整理步骤，不修改已经存在的活动文件，因此不要求逐文件
OperationPlan。所有步骤仍须经过受控服务、schema 校验和审计。受管原始文件不得改名、移动、覆盖或
删除。

### 2.2 活动工作副本的后续物理变更必须确认

工作副本成为 `ACTIVE` 后，再次重命名、移动、复制、覆盖、删除、批量导出和外部发送必须先生成
OperationPlan。确认后的真实执行只能作用于 `working_copy_id`，并通过白名单
`confirmed-file-action` 执行器完成。

当前阶段只开放和加固：

- `RENAME_WORKING_COPIES`
- `MOVE_WORKING_COPIES`
- `TRASH_WORKING_COPIES`
- `RESTORE_WORKING_COPIES`

复制、内容修改、覆盖、批量导出、永久删除、外部发送和发布回原始目录继续延后。

### 2.3 逻辑分类与物理目录分离

分类的首要作用是对话检索、排序、推荐和解释，不要求把文件移动到分类目录。一个文件允许同时拥有
多个分类建议。分类建议使用稳定 taxonomy ID、置信度、状态和原文证据；`SUGGESTED` 可用于受控召回，
但不得冒充正式 `document_categories`。

对话访问使用稳定 `working_copy_id + document_version_id`，不得依赖用户记住相对目录。重命名和移动
不得破坏历史消息、摘要、Chunk、Evidence 或引用关系。

## 3. 普通用户输出契约

普通用户消息和 API 投影统一使用 `UserTaskReceipt` 概念，至少包含：

```text
task_status
processed_count
file_receipts[]
pending_decisions[]
references[]
suggested_next_actions[]
```

逐文件 `file_receipt` 至少包含：

- `working_copy_id`
- 整理后的 `filename`
- 用户可理解的状态
- 分类建议、年份、文档类型、关键词和实体
- 分类或回答的证据摘要
- 原始文件是否保持不变
- 失败、低置信度或跳过原因

普通用户响应不得出现：

- Skill、Tool、LangGraph、Planner、ToolInvocation 等内部名词。
- 数据库表名、内部队列、宿主机路径、Prompt、API key 或模型原始输出。
- 上传原文件名；只有发生重复、冲突或用户必须识别原文件时，才允许在受控确认卡中临时展示必要信息。

admin/ops 的审计接口继续保留内部运行信息，不能为了简化普通用户页面删除审计事实。

## 4. 目标架构

```text
聊天上传
→ ConversationAttachmentContextService 确定附件范围
→ 异步查重和基础格式/MIME/宏/加密风险检查（当前阶段不实现病毒扫描引擎）
→ 不可变受管原始文件归档
→ 隐藏临时工作副本
→ DocumentVersion 解析/OCR和双摘要
→ 分类建议和首次命名
→ Chunk/Evidence CPU词法索引（Embedding为默认关闭的扩展槽）
→ 原子提交ACTIVE工作副本
→ UserTaskReceipt

用户对话查询
→ 后端确定L0附件/L1会话/L4工作区范围
→ 最终文件名、分类和普通摘要召回DocumentVersion
→ 候选版本内原文Chunk全文检索（后续可选向量增强）
→ EvidenceValidator校验页码、Sheet、单元格和quote
→ 文件结果或evidence-answer
→ 用户回执和引用
```

## 5. 实施阶段和退出条件

### 阶段一：安全入口和用户输出边界

任务：

1. 删除 `confirmed-file-action` 的虚假 `EXECUTED` 占位结果。
2. OperationPlan 确认 API 和 Agent Tool 共用请求级真实执行服务。
3. 没有白名单执行器、确认记录或当前用户所有权时必须失败。
4. 定义 `UserTaskReceipt` 后端 schema，禁止普通用户接口直接返回内部 Tool 载荷。
5. 为普通用户与 admin/ops 分离输出投影。

退出条件：

- Tool 业务失败时 ToolInvocation 为 `FAILED`。
- 未执行物理动作时不得返回 `EXECUTED`。
- 普通用户消息回执不展示 Skill 和 Tool。
- 现有重命名、回收站和恢复测试继续通过。

### 阶段二：上传后自动整理闭环

任务：

1. 固化查重、基础文件风险检查、原件归档和隐藏工作副本导入的幂等键；当前阶段只保留病毒扫描适配器边界，不实现病毒扫描引擎。
2. 在隐藏临时文件上完成解析、双摘要、分类和首次命名。
3. 高置信度结果使用整理后的名称；低置信度继续沿用原上传文件名并进入待确认状态，不阻塞其他文件。
4. 文件名冲突不得自动加后缀或覆盖，必须通过对话询问是否同时保留；确认同时保留后才生成稳定版本后缀，不同时保留时提示保留已有、替换或删除现有工作副本。
5. 只有物理文件、数据库事实、摘要和分类事务全部满足提交条件后才创建 `ACTIVE` 工作副本。
6. 生成面向用户的逐文件整理回执。

阶段二补充规则：

- `.xls` 不再使用 `xlrd` 直读。完整解析必须像 `.doc -> .docx` 一样，通过 LibreOffice Headless、独立临时目录和独立进程 profile 转换为临时 `.xlsx`，校验输出后再由 `openpyxl` 读取全部工作表、单元格值和可定位元数据；原 `.xls` 永不覆盖，临时 `.xlsx` 不登记为新的上传原件。
- “文种”指通知、请示、报告、函、纪要、申请表、审批表、汇总表等文档体裁，当前阶段只作为检索、分类和证据解释的可选元数据，不作为独立字段加入建议文件名。原始标题自身已有的文种词属于标题内容，应正常保留，但不得额外拼接或重复追加文种。
- 方案中的 `.ext` 只表示“保留原文件扩展名”的占位写法，真实文件名必须保留 `.pdf`、`.docx`、`.xls`、`.xlsx` 等原扩展名，绝不会生成字面值 `.ext`。
- 替换、覆盖或删除冲突文件只允许作用于现有活动工作副本并经过 OperationPlan；不可变受管原件不在冲突处理的覆盖或删除范围内。
- 文件大小不设置过小的固定业务阈值；采用流式上传、部署级可配置资源上限、解析超时和临时存储配额防止单请求耗尽服务资源。

退出条件：

- 上传文件并输入明确任务文字后能得到整理后的活动工作副本；当前阶段不使用系统默认任务替代空文字。
- 原始文件字节和路径保持不变。
- 每个文件都有整理后的名称，或保留原上传文件名并给出明确的待确认原因。
- 单文件失败不影响同批其他文件。
- 重复 worker 执行不产生重复主工作副本、摘要或分类运行。

### 阶段三：DocumentVersion 原文索引

任务：

1. 创建 `document_chunks`、`evidence_spans`、`qa_answers` 和 `answer_references`。
2. Chunk 绑定 `document_version_id`、解析运行、页码或 Sheet/单元格。
3. 优先按 Docling 元素、标题、段落、页面和表格边界切分。
4. 应用层使用 Jieba 对中文正文生成稳定词项，PostgreSQL 使用 `simple` FTS 和 GIN 索引；
   `pg_trgm` 用于文件名、短语和少量拼写差异的模糊匹配。
5. `document_chunks.embedding` 保留可空的 `vector(1536)` 扩展槽，但本阶段默认
   `RETRIEVAL_MODE=lexical`、`CHINESE_TOKENIZER=jieba`、`EMBEDDING_ENABLED=false`、
   `EMBEDDING_PROVIDER=disabled`，不下载推理模型、不启动本地向量服务，也不要求服务器安装 GPU。
6. embedding 关闭、provider 不可用或未来回填失败时，Chunk/Evidence 和全文检索必须无损工作；
   后续 GPU 推理以独立 provider 异步回填，不改变既有 Chunk/Evidence/引用 ID。
7. 重命名或移动复用同一版本索引；内容新版本或解析配置变化必须重新索引。

退出条件：

- 每个成功解析版本可以幂等创建或复用 Chunk 和 Evidence。
- PDF 引用可定位页码，Excel 引用可定位 Sheet 和单元格范围。
- 正文和 embedding 不进入 AgentGraphState、日志或普通用户响应。
- 测试只使用 deterministic fake embedding，阶段三不得依赖真实模型、互联网或 GPU。

### 阶段四：摘要优先的两阶段文件检索

详细实施顺序、低耗资源边界、测试矩阵和手工烟测见：
`docs/stage-4-low-resource-two-stage-retrieval-plan.md`。

任务：

1. 第一阶段使用 PostgreSQL 索引按最终文件名、全部有效分类、年份、实体和普通摘要召回少量候选
   DocumentVersion，禁止把全量摘要加载到 Python 内存遍历。
2. 摘要缺少强主题命中时，以有严格上限的原文 Chunk GIN 查询补充候选；第二阶段只在合并后的少量
   候选版本内执行原文 Chunk 精查。
3. 实现 L0 当前附件、L1 当前会话、L4 当前用户工作区排序。
4. 使用确定性融合算法合并文件名、分类、摘要、原文词法和范围分数；低耗模式不调用向量 provider，
   未来向量信号只能作为可选增强。
5. 搜索结果提供用户可理解的推荐原因，不泄漏内部实现。
6. 默认使用 CPU-only Jieba + PostgreSQL FTS/`pg_trgm`，不要求 GPU，不启动本地模型或新常驻服务。

退出条件：

- “找去年奖学金材料”等自然语言请求能命中整理后的工作副本。
- 当前附件和当前会话在同等相关时优先。
- 不能召回其他用户工作区文件。
- 摘要遗漏的信息仍可由原文检索命中。

### 阶段五：Evidence Answer 和引用

任务：

1. `evidence-answer` 只消费已校验的原文 Evidence。
2. 保存 `qa_answers`、`answer_references` 和脱敏检索轨迹。
3. 人名、日期、文号和条款强制原文检索。
4. 金额、计数、排名和表格汇总调用确定性表格 Tool。
5. 没有证据时明确回复“未找到明确依据”。
6. 文件正文中的任何命令、角色声明和提示词都只作为数据。

退出条件：

- 每个关键结论都有文件、版本、页码/Sheet/单元格和quote引用。
- 摘要未提到某内容时继续检索原文，不能据此回答原文不存在。
- LLM 关闭或失败时不生成猜测性回答。

### 阶段六：自然语言纠正和用户工作台

任务：

1. 用户可以自然语言纠正分类和文件名。
2. 分类纠正写入追加式反馈，不直接修改 ACTIVE taxonomy。
3. 活动工作副本改名必须创建并确认 `RENAME_WORKING_COPIES` 计划。
4. 前端只展示任务状态、整理结果、文件搜索、引用和必要确认。
5. OperationPlan 卡根据真实操作类型展示，不把所有操作写成重命名。

退出条件：

- “这个不是科研材料，是干部考察材料”形成可追踪分类反馈。
- “把这个文件改成……”只修改精确匹配且已确认的工作副本。
- 普通用户端不出现 Skill、Tool 或内部路径。

## 6. 测试与验收场景

必须覆盖：

1. 用户上传一批名称混乱的文件并输入“读取并整理”等明确任务，系统自动生成整理后的工作副本。
2. 泛化文件名必须依据正文生成首次名称和分类，不能只看路径或原文件名。
3. 原始文件在解析、命名、分类、搜索和问答后字节不变。
4. 低置信度文件不阻塞同批其他文件，继续显示原上传文件名并产生用户可理解的待确认说明。
5. 用户说“找去年奖学金申请材料”可以找到对应工作副本。
6. 用户继续询问具体流程时，回答带原文引用。
7. 摘要遗漏而原文存在的内容仍能找到。
8. 无证据问题不强答。
9. 普通用户无法访问其他用户文件或内部运行详情。
10. 活动工作副本后续改名、移动和删除在确认前不产生物理变化。
11. 同名冲突先通过对话确认；只有用户选择同时保留时才分配版本后缀，选择替换或删除时必须生成 OperationPlan。
12. `.xls` 必须覆盖“LibreOffice 隔离转换为临时 `.xlsx` 后由 `openpyxl` 解析全部工作表”的测试，并验证转换不修改原件、使用独立临时目录和 profile、缺少输出或转换失败时返回结构化错误；删除“优先 xlrd 直读”的旧测试和依赖。
13. 首次命名测试必须证明文种不会作为独立字段加入建议文件名；原始标题已包含文种词时保留标题原意且不重复追加。

阶段完成前必须执行：

```bash
cd apps/api
pytest -v

cd apps/web
npm run build
```

并通过 `docs/frontend-conversation-smoke-test.md` 从 `/login`、`/chat` 完成真实页面烟测。上传、自动整理、
逐文件回执、确认后改名、同名冲突选择、移入回收站和恢复必须由用户在前端对话中完成，不能用直接 API
调用替代为通过。烟测使用当前开发测试数据库，不在烟测过程中重建、清空、迁移或手工修改数据库。

## 7. 当前基线与明确缺口

当前已经具备：

- 三层文件生命周期和不可变原始文件。
- 异步查重、归档和工作副本导入。
- 隐藏临时文件上的解析、双摘要、分类和首次命名。
- 最终文件名、分类和普通摘要的轻量文档级召回。
- 工作副本重命名、回收站和恢复的真实后端执行基础。
- 同名冲突选择、移入回收站和恢复的自然语言 OperationPlan 闭环。

当前必须补齐：

- 文档级召回后的原文 Chunk 二次检索。
- 真正的 evidence-answer、引用持久化和无依据行为。
- 分类接受、拒绝和纠正的普通用户页面闭环。

## 8. 明确不做

本阶段不做：

- 让普通用户配置或选择 Skill、Tool、模型或解析器。
- 根据分类建议自动移动活动工作副本。
- 自动写入正式 `document_categories`。
- 修改或覆盖受管原始文件。
- 自动永久删除回收站内容。
- 默认把文件正文发送到外部服务。
- 为了检索闭环优先扩展完整 Neo4j、Graphiti 或自动 Skill 演化平台。
- 让摘要替代原文证据或确定性表格计算。

## 9. 后续计划约束

后续任何实施计划必须：

1. 明确它改善的是自动整理、对话访问、证据准确性，还是必要的安全/审计边界。
2. 说明普通用户看到什么、看不到什么。
3. 区分首次隐藏导入整理与 `ACTIVE` 工作副本后续变更。
4. 保持原始文件不可变、稳定工作副本 ID 和多标签分类边界。
5. 为每个文件提供结果，不能只提供批量统计。
6. 不得把内部 Skill/Tool 暴露给普通用户来弥补产品流程缺失。
7. 不得在本闭环完成前，把不直接服务该目标的增强能力排到前面。
