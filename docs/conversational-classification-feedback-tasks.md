# 对话式分类反馈与规则迭代开发任务

## 背景

后续分类升级不采用“每条分类都让用户点击确认”的表单式流程，而是保持聊天优先。系统生成的分类默认用于当前文件的检索、整理和展示；只有用户明确确认或明确纠正时，才记录为高权重学习信号。

## 产品原则

- 分类结果默认生效，但用户沉默只表示 `UNCONTESTED`，不能作为全局规则更新依据。
- 用户明确说“正确、都对”时，记录为 `CONFIRMED`。
- 用户明确说“第 N 个文件应归为……”或“不应归为……”时，记录为 `CORRECTED`，并先更正当前文件有效分类。
- 用户表达不完整时，进入 `NEEDS_CLARIFICATION`，由 Agent 追问，不自动猜测修改。
- 不允许 LLM 或用户自由文本直接新增全局分类节点；共享 taxonomy 只能通过候选、评估、审核、发布和回滚流程更新。

## 第一阶段最小闭环

1. 新增 `CLASSIFICATION_FEEDBACK` 意图识别，优先支持“第 N 个文件应归为 X”。
2. 新增 `FeedbackResolver`，只在当前 `user_id + conversation_id` 的最近分类结果中解析目标文件和分类路径。
3. 分类路径必须匹配当前 taxonomy 中已有节点；不唯一或不存在时返回澄清问题。
4. 成功解析后写入 `DocumentCategoryFeedback`，更新当前文件有效分类，并生成 `ChangeSet/ChangeItem`。
5. Agent 回执说明原分类、调整后分类、是否生成规则候选。
6. 前端只在批次结果末尾提示一次：“如分类不准确，可直接回复‘第 2 个文件应归为……’”，不增加确认按钮。

## 后续阶段

1. 扩展支持 `CONFIRM / ADD / REMOVE / 文件名引用 / 消歧追问`。
2. 新增 `taxonomy_change_candidates`，只从显式 `CONFIRMED/CORRECTED` 反馈生成候选。
3. 增加候选评估 CLI，输出新增命中、误判、冲突样本和净收益。
4. 增加 taxonomy 版本发布与回滚能力，历史分类运行保留原 `taxonomy_version`。
5. 可选增加用户级分类偏好层，只影响当前用户，不污染全局规则。

## 验收要求

- 用户无需按钮即可用自然语言纠正分类。
- 明确反馈最多一次澄清后完成当前文件分类更正。
- 所有反馈、分类变更、规则候选和 taxonomy 发布都可追溯。
- 未回复不会直接推动共享分类规则变更。
- 不允许跨用户、跨会话引用或修改文件分类。
