# File Rename Skill

## 触发条件

用户要求为聊天上传附件或服务器受管原始目录范围内的文件生成重命名建议，或确认此前生成的工作副本重命名计划。

## 输入输出 Schema

输入只允许后端已经解析并校验归属的上传附件 `document_ids`，或受管逻辑目录、完整相对目录前缀、扩展名和文件名过滤条件。后端必须把这些来源解析为当前用户活动的 `working_copy_id`；附件范围和受管目录过滤条件不得混用。输出包含年份、文号、正文标题、字段证据、建议文件名、待复核项、逐文件执行状态和 OperationPlan ID。

## Tool 白名单

- `generate-rename-suggestions`
- `confirmed-file-action`

## Open Source Backing

规则化命名和先预览后确认的思想参考 `tfeldmann/organize`。当前工作副本路径执行器为自研实现；历史 Native/F2 原地重命名执行器不再接入 Agent Runtime。

## 处理步骤

1. 确定重命名对象：上传附件使用明确 `document_ids`；受管原始文件使用逻辑目录和元数据过滤条件。两者都必须由后端解析为活动工作副本。
2. 异步归档或导入尚未完成时返回 `WAITING_FOR_ASYNC_JOB`；不得回退到上传暂存或受管原始目录执行。
3. 通过工作副本 Document 生成或复用 `document_pages`，读取完整正文。
4. 提取完整日期、年份、文号和正文标题，并保存证据。
   - `.xls` 必须由 LibreOffice 在隔离临时目录转为临时 `.xlsx`，校验后再由 openpyxl 读取；不得使用 xlrd 直读或覆盖原件。
   - `.xls/.xlsx/.xlsm/.csv/.tsv` 正文解析失败时，可从高结构化文件名回退；只清理表格文件名中的“附件”、处理日期、提交单位日期后缀、`new` 和“摸底统计表”噪声，并保留解析失败审计。
5. 使用 `_` 生成 `{year}_{document_number}_{title}`；普通材料缺文号时降级为 `{year}_{title}`。
6. 只把 `READY` 项加入 `RENAME_WORKING_COPIES` OperationPlan；`NEEDS_REVIEW` 和冲突项不得执行。
7. OperationPlan 由后端写入当前相对路径、当前 DocumentVersion、内容 SHA-256 和工作副本路径记录，用户确认后才能执行。
8. 执行器只改变工作副本路径、工作副本展示名和同一版本的存储路径，不创建新版本，不修改受管原始文件。
9. 自动建议的基础目标名称已存在时，保留新文件的原上传文件名并请求用户选择；只有用户明确要求同时保留后，后续确认流程才能分配稳定版本后缀。
10. 同一目录同一批次出现相同标题时，先使用正文中的 `YYYYMMDD` 日期区分；日期相同或缺失时再使用版本号。

## 失败与降级

- 缺少年份或标题：`NEEDS_REVIEW`，不进入执行批次。
- 上传附件或受管范围尚未建立工作副本：`WAITING_FOR_ASYNC_JOB`，不修改任何文件。
- 范围无法唯一解析：保留 `NEEDS_REVIEW` 并要求用户提供完整相对目录或稳定文件选择。
- 缺少文号：普通材料使用降级模板。
- 自动建议目标冲突：不得自动加后缀或覆盖；请求用户选择同时保留、保留已有、替换现有工作副本或删除现有工作副本。
- 用户手工指定目标冲突、源文件变化、路径越界或目录未授权：拒绝执行当前项，其他项继续。
- 失败项必须写结构化错误和 ChangeItem，不得伪造成功。

## 验收标准

- 确认前工作副本不变，确认前后受管原始文件始终不变。
- 普通用户只能确认自己的计划。
- 执行后工作副本索引路径与文件系统一致，且 DocumentVersion 数量不增加。
- 每个重命名项都存在带执行时间的工作副本路径记录。
- 每个成功项存在 `FILENAME_CHANGED`，每个失败项存在失败审计。

## 禁止事项

- 不允许绕过 OperationPlan 重命名工作副本；不允许修改上传暂存或受管原始目录。
- 不允许 LLM 提供绝对路径或直接执行文件操作。
- 不允许覆盖已存在目标文件。
- 不允许执行 `NEEDS_REVIEW` 项。
- 不允许把用户提供的名称直接传给文件系统；必须先固化 OperationPlan 和确认记录。
- 不允许 LLM 或用户提供 Shell 命令或绝对路径。
