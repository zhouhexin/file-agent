# File Rename Skill

## 触发条件

用户要求为服务器受管目录中的文件生成重命名建议、确认此前生成的重命名计划，或为待复核文件明确提供更正后的名称。

## 输入输出 Schema

输入只允许受管逻辑目录、相对目录前缀、扩展名、文件名过滤条件，以及“文件原文件名更正为新文件名”的明确更正消息。输出包含年份、文号、正文标题、字段证据、建议文件名、待复核项、逐文件执行状态和 OperationPlan ID。

## Tool 白名单

- `generate-rename-suggestions`
- `resolve-rename-reviews`
- `confirmed-file-action`

## Open Source Backing

规则化批处理思想参考 `tfeldmann/organize`。项目保留自研 Native 执行器，并提供 F2 v2.2.2 受控 Adapter；F2 只执行后端已经固定的 `before -> after` 映射，不读取正文、不生成名称，也不能绕过受管目录、OperationPlan 和 ChangeSet。

## 处理步骤

1. 确定性解析受管目录范围和元数据过滤条件。
2. 创建或复用不可变快照，从 `document_pages` 读取完整正文。
3. 提取完整日期、年份、文号和正文标题，并保存证据。
   - `.xls` 优先由 `xlrd>=2.0.1` 直接读取，失败后才尝试可选 LibreOffice 转换；当前不强制安装 LibreOffice。
   - `.xls/.xlsx/.xlsm/.csv/.tsv` 正文解析失败时，可从高结构化文件名回退；只清理表格文件名中的“附件”、处理日期、提交单位日期后缀、`new` 和“摸底统计表”噪声，并保留解析失败审计。
4. 使用 `_` 生成 `{year}_{document_number}_{title}`；普通材料缺文号时降级为 `{year}_{title}`。
5. 只把 `READY` 项加入 OperationPlan；`NEEDS_REVIEW` 和冲突项跳过原批次，并按用户、会话和受管文件持久化。
6. 用户可点击待复核文件查看原件；回执不逐项公开内部缺失字段，只提示按“文件原文件名更正为新文件名”回复或回复“不需要”。
7. 用户明确提供新名称时，将该消息作为执行确认，仍需先创建 OperationPlan 和 OperationConfirmation，再由 Native 或 F2 执行器执行并写 ChangeSet。
8. 自动建议的基础目标名称已存在时，基础名称视为第一版，依次生成 `_第二版`、`_第三版`；同一批次内的目标名也必须参与版本分配。
9. 同一目录同一批次出现相同标题时，先使用安全快照原始文件名或正文中的 `YYYYMMDD` 日期区分；日期相同或缺失时再使用版本号。
10. 用户手工更正的原文件名匹配多个待复核项时列出完整相对路径；手工指定目标名称重复时只跳过冲突项，其他唯一项继续执行。
11. F2 模式必须先 dry-run，输出与 OperationPlan 完全一致后才允许增加 `-x`。

## 失败与降级

- 缺少年份或标题：`NEEDS_REVIEW`，不进入执行批次。
- 用户回复“不需要”：将当前会话待复核项标记为 `DISMISSED`，不修改文件。
- 原文件名匹配多个待复核项：保留 `NEEDS_REVIEW` 并返回完整相对路径候选。
- 缺少文号：普通材料使用降级模板。
- 自动建议目标冲突：按中文版本后缀分配下一个可用名称；达到版本上限时进入待复核。
- 用户手工指定目标冲突、源文件变化、路径越界或目录未授权：拒绝执行当前项，其他项继续。
- F2 缺失、版本不一致、超时或返回未知 JSON：整个相关批次拒绝执行，不默认静默回退。
- 失败项必须写结构化错误和 ChangeItem，不得伪造成功。

## 验收标准

- 确认前源文件不变。
- 普通用户只能确认自己的计划。
- 执行后索引路径与文件系统一致。
- 每个成功项存在 `FILENAME_CHANGED`，每个失败项存在失败审计。

## 禁止事项

- 不允许重命名上传原件。
- 不允许 LLM 提供绝对路径或直接执行文件操作。
- 不允许覆盖已存在目标文件。
- 不允许执行 `NEEDS_REVIEW` 项。
- 不允许把用户更正直接传给文件系统；必须先固化 OperationPlan 和确认记录。
- 不允许 LLM 或用户提供 F2 参数、CSV、Shell 命令或绝对路径。
- 不允许 F2 使用覆盖、自动修复冲突、隐藏文件、目录改名、跨目录移动或全局 undo 参数。
